你现在让 AI 干活，多半是这样：丢一个复杂任务过去，他自己在后台调起一堆工具，一会儿就把成品端给你。最近他这身本事还在变强，比如 Claude Code 里的 Agent Skill（智能体技能）。

这当然省心。但他越是能一手包办，你反而越像在开盲盒：东西交到你手上之前，他在里面做了什么、对不对，你一概不知。

我们在做一项30天公开构建实验，用 Playtiss 重构一个让人和 AI 对等协作的 Todo Map。今天聊个反方向的问题：AI 的 Skill 越来越能一手包办，我们为什么还要把事情拆成一个个工作流的节点？

## 黑盒里的天才员工

想象一下，你招了一个极其聪明的“天才员工”。你交给他一个棘手的任务，他二话不说一头扎进办公室，“砰”地关上门，在里面忙了一整天。

如果他中途遇到一个拿不准的数据，他只能拉开一条门缝，问你一个孤立的问题。如果最后他拿出来的方向全错了，你根本不知道他是在哪一步跑偏的：你看不到他的草稿，也没法让另一个同事进去，接手他做到一半的工作。

这就是 Agent Skill 在真实协作里的困境。Skill 内部就是一个高度线性的黑盒。跑得顺的时候，他在里面一路调工具、一路往下做，灵活又高效。

可他一旦出错、产生幻觉，或者走到一个需要人来拿主意的岔口，黑盒就只能停下来，跟你一问一答。最麻烦的是，东西全在黑盒里跑，你没法塞进第二个人，或者第二个 AI，去独立查一查他中间那几步到底对不对。

于是你又被钉回了昨天那个位置：疲于奔命的操作员。他在黑盒里闷头跑，你守在门外，等他探头、替他拿主意，还没人能搭把手。

## 审计也是一种协作

退一步想：真实工作里，我们为什么要把一个大任务拆成一步步？

不只是为了好执行，更是为了留下责任的边界。

要查一步做得对不对，你得能单独看到这一步的输入和产出。黑盒两样都不给你，只吐出一个最终结果。把活拆成一步步，“检查”这件事才重新变得可能。

我们在《协作协议》里有一条：审计和追责，本质上就是一种协作。它是让另一个独立的人，或者另一个不同的 AI，去检查上游那一步的结果，再给出反馈。比如让擅长写代码的 Claude 去开发，让擅长挑逻辑的 Gemini 去核查。

如果太迷信 Agent Skill，把整条流程都塞进一个大黑盒，这种跨 AI 的交叉检查就被彻底堵死了。

这也是为什么那些大公司的企业级软件，到今天还死抱着“工作流”不放：他们需要边界，需要留下一条清清楚楚、能回头追溯的痕迹。

## 把黑盒关进边界里

那是不是又得回到昨天说的老路，先画一张死板的流程图才能开工？

当然不用。Playtiss 想做的，是让工作流和 Agent Skill 各管一段。

这正是前两天聊过的两件事：让工作流“从 0 到 1”几乎没有建立成本（Day 2），让“和 AI 一起改图”几乎没有维护成本（Day 3）。你照样从一个意图起步，让网络自行裂变出清晰的节点。

然后，你把最核心、最繁杂的那一个节点，整个放权给带 Agent Skill 的 AI，让他在里面“黑盒作业”，尽管发挥。

但他做完，产出必须走出黑盒，停在 Playtiss 的边界上。在这个边界，你可以放一个审查节点：你自己看一眼，或者让另一个 AI 来交叉比对。

在这里，这一个节点的输入和产出都是干净、独立的。你（或另一个 AI）不必查看黑盒里的全过程，只要看一眼“给了这个、出了那个，对不对”。觉得不对，就把这一步打回去重跑，后面没动过的节点不受影响。

*[图例：一条 Playtiss 工作流（DAG）。中间一个节点是黑色的“黑盒”，AI 在里面用 Skill 自由作业；它的输出连到下一个亮色的“审查”节点，节点旁站着一个人和另一个 AI。]*

宏观上，Playtiss 这张网络负责划分责任、防止一处出错就全盘崩、让每一步的输入输出都看得见。微观上，在每一个节点内部，你尽管让 AI 的 Skill 发挥它的灵活和高效。

这才是看得见、管得住、也容得下错的协作。

那如果有一天，单个大模型真聪明到能“一手包办、永不出错”，这张网络还有没有用？这个更长远的问题，我们明天接着聊。
