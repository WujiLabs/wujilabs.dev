前两天，我想改 Todo Map 里一个牵动好几个模块的功能。大模型很认真地警告我：这要动几十个文件，一个人来做，得半年以上。

我没理他，让他直接开干。结果大概 10 个小时，全写完了。

一个 10 小时能干完的活，他为什么张口就要半年？说穿了，他是在拿人的工期，去量自己的活；这两笔账，根本不是一回事。

今天是 Todo Map（待办拓扑图）公开构建的第 8 天。想跟你分享一个我反复撞见的连锁反应：无论是做这个 App，还是更早做 Playtiss 引擎，只要项目一上规模，他在“规划和调度”上就会接连出岔子。开头那个“半年”的误判，只是其中最显眼的一环。

## 计划崩溃的连锁反应

我开发时的搭配很简单：Claude Code 主力写代码，Codex 在旁边帮我审查。

当一个项目从模糊的想法，长成几十个文件的系统，你会攒下越来越多“想留住的好设计”和“得绕开的坑”。这些东西没法写成测试自动去查，只能当成一条条规则，堆进他的上下文里。受上下文的限制，规则一多，反应链就开始了。

**第一环：计划拉长，要求漏项。**
规则多了，你要是让他拟一份详尽的开发计划，计划就会变长。一长，他就会在中间漏掉某些具体要求。这是注意力机制的硬限制，不是他不上心。

**第二环：计划缩短，“推迟”变“遗忘”。**
为了不漏要求，他会本能地把计划压短，建议把一些功能“往后放”。
大模型其实有自己的记忆文件，也有待办列表。但麻烦在于，一个任务一旦被丢进待办列表里往后放，往往就成了“无限期往后放”。项目收尾时，100 个任务做完了 95 个，剩下的就这么静静地死在记忆文件里。

**第三环：荒谬的“推迟”标准。**
他凭什么决定哪个任务往后放？很大程度上，靠的就是开头那把“人的尺子”。他看一个改动牵动多少文件、动几个模块，数字越大，就越判成“多季度工程”，越该往后挪。
可那把尺子量的是人：人要开会、要上手、改完还得等代码评审。他没这些开销，涉及的代码都装在上下文里，写、测、自审一口气做完，每步成本小上一两个数量级。
文件数是真的，只是它跟他真正花的时间几乎不挂钩。于是那些顺手就能做掉的事，被他按人类的工期，判了个“无限期缓刑”。

**第四环：到极限处，动作变形。**
哪怕你把计划规模控制得很好，单次会话的上下文快满时，他照样会出错。
几个月前，上下文快满时，大模型往往草草把活干一半，就宣布“做完了”。最近的模型改了做法，会直接停下来，让你清理上下文再喊他继续。形式怎么变都行，但逼近记忆极限时，他遵循计划的能力会陡然下滑。

## 我现在的应对：硬性分级隔离

面对这些瓶颈，在真正好用的自动化引擎成熟之前，我现在的“人工补丁”是做硬性的分级计划和隔离：

1. **宏观和微观分开。** 一开始只让他定一个总体的、分几个 Phase（阶段）的粗略计划。绝不让他一上来就把所有细节想透。
2. **锁定单个 Phase。** 真正进到某个 Phase，再逼他针对这一阶段定详尽的微观计划。
3. **强制交叉审查。** 一个 Phase 做完，绝不直接进入下一步。一定引入另一个模型来审（比如用 Codex 审 Claude 的代码），再手动核对当初的待办列表，防漏项。
4. **手动压缩、清空。** 全部确认无误，手动压缩上下文（或者用 `retcon` 开一个干净的会话），再进下一个 Phase。
5. **时间估算我自己来。** 遇到要评估时间或复杂度的时候，一律我自己判断；或者干脆让他先别管时间，直接动手。

## 把记忆还给拓扑图

上面这套管用，但确实很累。

最近 Claude Code 也推出了动态工作流（Dynamic Workflows）。它的思路其实和我想的是一个方向：把编排逻辑从对话上下文里挪出去，写成一段脚本，让脚本自己保管循环、分支和中间结果，大模型的上下文只留最后的答案。

这恰好说明，“别把计划全压在单个大模型的上下文里”，已经在变成行业共识。只是动态工作流主要解决单次大任务里的并行编排，脚本跑完，这一趟的中间状态也就散了。

我更想往前走一步。这也是我们做 Playtiss 工作流引擎时一直在啃的边界问题：能不能把静态的“待办清单”，变成一张客观存在于外部的拓扑图，让一个被推迟的任务，不至于烂在某段上下文、某个文本文件里？

我不确定 Playtiss 就是这个问题的终极答案，但这确实是它的设计初衷：被推迟的任务挂在外部工作流的一个节点上，等前置条件一满足，或者大模型切进一个干净的新环境，引擎就自动把它重新推回来。

把宏观的责任调度交给外部的基底，把最纯粹的算力留给当下。这或许才是跳出“上下文焦虑”的出路。
